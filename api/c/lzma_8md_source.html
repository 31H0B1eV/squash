1 # lzma Plugin #
2 
3 The lzma plugin provides an interface for liblzma from the [XZ Utils
4 project](http://tukaani.org/xz/).
5 
6 ## Codecs ##
7 
8  * **xz**: The stream format used by default in the xz tool.
9  * **lzma**: Legacy format used by the lzma tool. You should use xz
10  instead unless you need this for compatibility reasons.
11  * **lzma2**: Format with a single-byte header used by xz. Requires
12  *dict-size* to be present for the decoder. Relying on the default
13  *value is dangerous as it can change from version to version.
14  * **lzma1**: Raw format used by lzma. The following options should
15  be present: *dict-size*, *lc*, *lp*, *pb*. Relying on default
16  values for those options is dangerous as those defaults can change
17  from version to version.
18 
19 ## Options ##
20 
21  * **level** (integer, 1-9, default 6): Set the compression level. 1
22  will result in the fastest compression while 9 will result in the
23  highest compression ratio.
24  * **check** (enumeration, default crc64, xz only): Set the algorithm
25  used to verify the compressed data. Available values:
26  * *none*: do not verify
27  * *crc32*: [CRC](https://en.wikipedia.org/wiki/Cyclic_redundancy_check)-32, suitable mainly for small files.
28  * *crc64*: [CRC](https://en.wikipedia.org/wiki/Cyclic_redundancy_check)-64, suitable for larger files.
29  * *sha256*: [SHA-256](https://en.wikipedia.org/wiki/SHA-2), for
30  when security is important.
31  * **dict-size** (integer, 4096-1610612736 (2^30 + 2^29), default
32  8388608 (2^23), required for lzma1 and lzma2): From the xz man
33  page:
34  > Dictionary (history buffer) size indicates how many bytes of the
35  > recently processed uncompressed data is kept in memory. The
36  > algorithm tries to find repeating byte sequences (matches) in the
37  > uncompressed data, and replace them with references to the data
38  > currently in the dictionary. The bigger the dictionary, the
39  > higher is the chance to find a match. Thus, increasing
40  > dictionary size usually improves compression ratio, but a
41  > dictionary bigger than the uncompressed file is waste of
42  > memory.
43  >
44  > Typical dictionary size is from 64 KiB to 64 MiB. The minimum is
45  > 4 KiB. The maximum for compression is currently 1.5 GiB (1536
46  > MiB). The decompressor already supports dictionaries up to one
47  > byte less than 4 GiB, which is the maximum for the LZMA1 and
48  > LZMA2 stream formats.
49  >
50  > Dictionary size and match finder (mf) together determine the
51  > memory usage of the LZMA1 or LZMA2 encoder. The same (or bigger)
52  > dictionary size is required for decompressing that was used when
53  > compressing, thus the memory usage of the decoder is determined
54  > by the dictionary size used when compressing. The .xz headers
55  > store the dictionary size either as 2^n or 2^n + 2^(n-1), so
56  > these sizes are somewhat preferred for compression. Other sizes
57  > will get rounded up when stored in the .xz headers.
58  * **lc**: (integer, 0-4, default 3, required for lzma1): From the xz
59  man page:
60  > Specify the number of literal context bits. The minimum is 0 and
61  > the maximum is 4; the default is 3. In addition, the sum of lc
62  > and lp must not exceed 4.
63  >
64  > All bytes that cannot be encoded as matches are encoded as
65  > literals. That is, literals are simply 8-bit bytes that are
66  > encoded one at a time.
67  >
68  > The literal coding makes an assumption that the highest lc bits
69  > of the previous uncompressed byte correlate with the next byte.
70  > E.g. in typical English text, an uppercase letter is often
71  > followed by a lower-case letter, and a lower-case letter is
72  > usually followed by another lower-case letter. In the US-ASCII
73  > character set, the highest three bits are 010 for upper-case
74  > letters and 011 for lower-case letters. When lc is at least 3,
75  > the literal coding can take advantage of this property in the
76  > uncompressed data.
77  >
78  > The default value (3) is usually good. If you want maximum
79  > compression, test lc=4. Sometimes it helps a little, and
80  > sometimes it makes compression worse. If it makes it worse, test
81  > e.g. lc=2 too.
82  * **lp** (integer, 0-4, default 0, required for lzma1): From the xz
83  man page:
84  > Specify the number of literal position bits. The minimum is 0
85  > and the maximum is 4; the default is 0.
86  >
87  > Lp affects what kind of alignment in the uncompressed data is
88  > assumed when encoding literals. See pb below for more
89  > information about alignment.
90  * **pb** (integer, 0-4, default 2, required for lzma1): From the xz
91  man page:
92  > Specify the number of position bits. The minimum is 0 and the
93  > maximum is 4; the default is 2.
94  >
95  > Pb affects what kind of alignment in the uncompressed data is
96  > assumed in general. The default means four-byte alignment
97  > (2^pb=2^2=4), which is often a good choice when there's no better
98  > guess.
99  >
100  > When the aligment is known, setting pb accordingly may reduce the
101  > file size a little. E.g. with text files having one-byte
102  > alignment (US-ASCII, ISO-8859-*, UTF-8), setting pb=0 can improve
103  > compression slightly. For UTF-16 text, pb=1 is a good choice.
104  > If the alignment is an odd number like 3 bytes, pb=0 might be the
105  > best choice.
106  >
107  > Even though the assumed alignment can be adjusted with pb and lp,
108  > LZMA1 and LZMA2 still slightly favor 16-byte alignment. It might
109  > be worth taking into account when designing file formats that are
110  > likely to be often compressed with LZMA1 or LZMA2.
111 
112 ### Decoder-only ###
113 
114  * **memlimit** (integer, default UINT64_MAX): Memory limit to use
115  while decoding. Note that decoding could fail if this is set too
116  low.
